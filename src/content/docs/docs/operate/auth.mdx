## The default posture

hal0 v0.3 binds the API on `0.0.0.0:8080` with **no built-in
authentication**. The old Caddy / Bearer / `--auth=basic` machinery
was removed in v0.3.0-alpha.1 per **ADR-0012**.

For a LAN-only single-user install — an LXC on your Proxmox node
that nothing outside your house can reach — that's fine and it's
the intended posture. For everything else (a port-forward, a
Tailscale exit that other people share, a public hostname, a guest
VLAN), put hal0 behind a reverse proxy and do auth there.

<Aside type="caution" title="LAN vs. WAN">
On a trusted LAN with no port forwards, the no-auth default is
fine. If hal0 is reachable from anything you don't physically
control, put it behind a reverse proxy. There is no middle ground —
and there is no longer a built-in `--auth=basic` to fall back on.
</Aside>

## Reverse-proxy recipes

Three minimal examples. Pick the one that matches the proxy you
already run.

<Tabs>
<TabItem label="Caddy">

```caddy
# /etc/caddy/Caddyfile — single vhost, basic_auth, automatic HTTPS.
hal0.example.com {
  basic_auth {
    alex {$HAL0_ADMIN_BCRYPT}
  }
  reverse_proxy 127.0.0.1:8080
}
```

Caddy handles ACME against Let's Encrypt automatically.

</TabItem>
<TabItem label="Traefik">

```yaml
# /etc/traefik/dynamic/hal0.yml — single dynamic config snippet.
http:
  routers:
    hal0:
      rule: "Host(`hal0.example.com`)"
      service: hal0
      entryPoints: [websecure]
      tls: { certResolver: letsencrypt }
      middlewares: [hal0-auth]
  middlewares:
    hal0-auth:
      basicAuth:
        users: ["alex:$2y$05$..."]   # htpasswd bcrypt
  services:
    hal0:
      loadBalancer:
        servers: [{ url: "http://127.0.0.1:8080" }]
```

</TabItem>
<TabItem label="nginx">

```nginx
# /etc/nginx/sites-enabled/hal0 — server block with basic_auth.
server {
  listen 443 ssl;
  server_name hal0.example.com;
  ssl_certificate     /etc/letsencrypt/live/hal0.example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/hal0.example.com/privkey.pem;
  auth_basic           "hal0";
  auth_basic_user_file /etc/nginx/hal0.htpasswd;
  location / {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
  }
}
```

You bring the TLS cert (certbot or your own ACME client).

</TabItem>
</Tabs>

## Identity in the upstream

hal0 still has the concept of an **agent identity** vs. a **user
identity**. That distinction now rides on a header instead of a
token: agents and CLIs send `X-hal0-Agent: <agent_id>`; the absence
of the header is treated as "user".

If your reverse proxy needs to identify the caller — for example,
to inject a per-user identity that hal0 then uses for memory
namespacing or audit logging — set the `X-hal0-Agent` header
server-side in the proxy. Strip any inbound copy of the header
first (so a caller can't spoof one) and then inject the value you
trust.

See [Agent identity](/docs/agents/identity/) for the contract,
including the outstanding server-side wiring (the header read
path is still landing).

## What was removed

For users migrating from v0.1 / v0.2: ADR-0012 removed **all** of
the built-in auth surface in v0.3.0-alpha.1. Specifically gone:

- The bundled **Caddy** reverse proxy and `hal0-caddy.service`.
- **Bearer tokens** in `/etc/hal0/tokens.toml` and the `/api/auth/tokens`
  CRUD endpoints.
- The `--auth=basic` installer flag, `HAL0_AUTH_ENABLED` /
  `HAL0_AUTH_DISABLED` env, the OTP lockfile, and the password
  claim flow.

If you were using any of the above, the v0.3 migration path is
"front hal0 with your own reverse proxy" using one of the recipes
above. Existing trusted-LAN installs with no auth keep working
unchanged.
