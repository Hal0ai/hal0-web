In v0.3, hal0 removed all built-in auth (ADR-0012; the dashboard and
API bind `0.0.0.0:8080` open). What replaces it for agent-vs-user
disambiguation is a single header: `X-hal0-Agent`. Every agent
process is started with a `HAL0_AGENT_ID` environment variable, and
the agent runtime sets the header on every outbound HTTP and MCP
request.

## The contract

| Direction                      | Contract                                                                       |
|--------------------------------|---------------------------------------------------------------------------------|
| Agent → hal0-api (HTTP / `/v1/*`) | `X-hal0-Agent: <agent_id>` on every request.                                |
| Agent → hal0 MCP servers       | Same header on the MCP `Streamable-HTTP` connect.                              |
| User → hal0 (browser, CLI)     | No header. The server treats this as the "user" identity.                       |
| Reverse proxy in front of hal0 | Inject `X-hal0-Agent` server-side if you need richer identity. Strip inbound copies first. |

`HAL0_AGENT_ID` is set by the systemd unit (e.g. `hal0-hermes.service`
sets `Environment=HAL0_AGENT_ID=hermes-agent`). Agent runtimes never
derive the id themselves; they read it from the env at startup and
keep it for the process lifetime.

## What the server does with it

Today: the API mounts the header into the request context and
journals it on every approval, every memory write, every MCP tool
call. Agents-vs-user disambiguation in logs and the agent inbox
journal works.

What's still landing — see the caution block below.

## Why a header, not a token

Auth in v0.3 lives **at the edge** (your reverse proxy, your LAN
posture, your homelab gateway). hal0 no longer mints, stores, or
validates secrets. That means agent identity has to be a thing the
proxy can inject and the server can trust within its trust boundary,
not a thing that has to round-trip through a credential store.

A plain header satisfies that:

- Reverse proxies can override it server-side (strip inbound, inject
  the real value) the same way they'd handle `X-Forwarded-For`.
- The dashboard's "user" identity is just the absence of the header.
- Agents don't have to bootstrap a credential before they can call
  hal0.

See [Authentication](/docs/operate/auth/) for the reverse-proxy
recipes.
